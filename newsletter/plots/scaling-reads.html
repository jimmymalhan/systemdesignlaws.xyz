Indexing: Full Scan vs B-Tree Lookup

Turn O(n) into O(log n) - the single biggest read optimization

Without Index (Full Table Scan)
Scans ALL 10M rows
100-500ms latency
CPU at 100%
O(n) - linear scan
No write overhead
With B-Tree Index
Traverses 3-4 tree levels
1-5ms latency
Minimal CPU usage
O(log n) - tree lookup
Small write cost per index

Left: sequential scan reads every row. Right: B-tree jumps directly to the match in 3-4 hops.

-- Before: Full table scan (10M rows)
EXPLAIN SELECT * FROM users WHERE email = 'user@example.com';
-- Seq Scan on users (cost=0.00..412,000.00 rows=1)

-- Add index
CREATE INDEX idx_users_email ON users(email);

-- After: Index scan (instant)
EXPLAIN SELECT * FROM users WHERE email = 'user@example.com';
-- Index Scan using idx_users_email (cost=0.43..8.45 rows=1)

Denormalization: Trade Storage for Speed

Eliminate joins by storing redundant data

Normalized (3-Table JOIN)
users + orders + products
Multiple disk seeks per query
Clean data model
Single-point writes
10-50ms per read
Denormalized (Single Table)
order_summary has everything
One row = one disk read
Redundant data stored
Multi-point writes
1-5ms per read
-- Normalized: expensive on every page load
SELECT p.id, AVG(r.rating) as avg_rating
FROM products p JOIN reviews r ON p.id = r.product_id
GROUP BY p.id;

-- Denormalized: precompute once
CREATE MATERIALIZED VIEW product_ratings AS
SELECT p.id, AVG(r.rating) as avg_rating
FROM products p JOIN reviews r ON p.id = r.product_id
GROUP BY p.id;

Cache Stampede: The Thundering Herd

When a popular cache key expires, thousands of requests hit the DB simultaneously

Left: stampede when TTL expires. Right: probabilistic refresh spreads the load. Most users still get cached data.

# Request coalescing - combine duplicate requests
class CoalescingCache:
  def __init__(self):
    self.inflight = {} # key -> Future

  async def get(self, key):
    if key in self.inflight:
      return await self.inflight[key]

    future = asyncio.Future()
    self.inflight[key] = future
    try:
      value = await fetch_from_backend(key)
      future.set_result(value)
      return value
    finally:
      del self.inflight[key]
# Reduces backend load to N (one per app server)

Cache Versioning: Sidestep Invalidation

Don't delete old cache entries - make them irrelevant by changing the key

Version changes are atomic in the database. Cache keys automatically route to the new version. Old entries expire naturally.

-- On write: increment version atomically
BEGIN;
UPDATE events SET venue = 'New Location', version = version + 1
  WHERE id = 123;
COMMIT;

-- Cache keys before/after:
event:123:v42 // old - still in cache but no one reads it
event:123:v43 // new - app reads this now
